# [Code Review] - [MEDIUM] - Dead code: unused `_api` variable and `ToolCallDelta` event

**Labels:** `tech-debt`, `priority-medium`, `dead-code`

## Description

Two pieces of dead code exist in the project:

1. **Unused `_api` client in `main.rs:60`**: `let _api = ApiClient::new(ApiConfig::default());` creates a client that is never used. A second client is created later on line 145. The prefix `_` suppresses the unused-variable warning, but this is misleading — the variable should either be removed entirely or used.

2. **`TuiEvent::ToolDelta` is sent but never consumed**: `main.rs:80-86` sends `TuiEvent::ToolDelta` with `id: String::new()` (empty ID), but `app.rs:102` handles this event with `TuiEvent::ToolDelta { id: _, delta: _ } => {}` — an empty match arm that does nothing. No data is ever stored or rendered from this event. The `AgentEvent::ToolCallDelta` variant in `agent.rs` is generated by the agent but never actually emitted (no code creates `AgentEvent::ToolCallDelta`), so this event chain is entirely dead.

## Location

- `hackpi-tui/src/main.rs:60` — Unused `_api`
- `hackpi-tui/src/main.rs:80-86` — `ToolCallDelta` sender with empty id
- `hackpi-tui/src/app.rs:102` — Empty handler for `ToolDelta`
- `hackpi-core/src/agent.rs:14` — `ToolCallDelta` variant never constructed

## Impact

- Confusing code: new developers will wonder why there are two clients and what `ToolDelta` does
- Wasted maintenance surface — 4 variants/lines of code that do nothing
- The empty `id` in `ToolCallDelta` suggests the implementation was never finished

## Proposed Solutions

1. Remove `let _api` from main.rs:60
2. Either implement `ToolCallDelta` properly (streaming tool output to TUI in real-time) or remove the entire variant chain from `AgentEvent`, `TuiEvent`, `app.rs`, and `main.rs`
3. Remove the unused `AgentEvent::ToolCallDelta` from `agent.rs` if it's never constructed
